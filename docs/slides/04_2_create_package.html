Create packages¶

Package tools¶

It is possible to create package in different ways

setup.py¶

Classic way, you write everything in the setup.py. Where is only one tool that can package it and it is distutils. Recommendation for packaging projects, description of distutils features.

PEP-517/PEP-518¶

New configuration file is added: pyproject.toml Now, you can specify dependencies that required for installation. They are used during installation and removed after (PEP-518).

You can specify witch tool should be used for installation (PEP-517). You are not tied to distutils anymore. This opens a new era in python distribution.

Here is incomplete list of tools that can be used:

distutils¶

Default tool, still requires setup.py to create a package.

poetry¶

All package data is located in pyproject.toml, no need to have setup.py anymore. Besides from packaging this tool focuses on dependency management.

flit¶

All package data is located in pyproject.toml, no need to have setup.py anymore. Cannot be used when need to compile C or bundle Javascript. This tool has good support for publishing on PyPI.

Information required for packaging¶

  • package meta info: name, description, version etc. If you publish your library somewhere make sure that you fill this field with normal data. Use semantic versioning

  • which python is supported by your package. Python 3.5 is a good start for public libraries, for limited number of users try to use as new version as you can. It will allow you to use all new shiny features

  • required dependencies for your package. Don’t forget to specify exact versions or at least ranges

  • script that will be installed to system (like pip). You can have own script that will be copied to executable dir distutils or use python function that will be packed as script (Entry point) distutils poetry
